## 3. Put it into production (your call — this DEPLOYS)

`main` auto-deploys to the VPS via the deploy poller, so merging to `main` ships to production. Do this only after the UI check above.

1. Decide the integration target (usually `main`).
2. Merge the combined branch:
   ```
   git checkout main && git pull
   git merge feat/fms-action-nodes
   ```
3. **Expect conflicts** on files that both this work and the concurrent delegation work touched:
   `backend/main.py`, `backend/database.py`, `backend/scheduler.py`,
   `frontend/src/lib/api.js`, `frontend/src/App.js`,
   `frontend/src/components/layouts/AdminNavItems.js`,
   `frontend/src/components/fms/FlowFormDialog.js`.
   These are almost all **additive** (new router include, new index, new loop, new API methods, new nav entry) — resolve by keeping **both** sides. Ask me to do the merge and I'll resolve them.
4. **Set `PUBLIC_BASE`** in the backend environment to your public HTTPS base URL (e.g. `https://app.smartshape.in`). WhatsApp fetches certificate PDFs from `PUBLIC_BASE/uploads/certificates/...`; without it, WhatsApp delivery of certificates fails.
5. Ensure `backend/uploads/certificates/` exists and is writable on the VPS (served as a static mount).
6. After deploy, send ONE real test certificate to your own number/email before any bulk run.

## 4. Known constraints (carry forward)

- **Single backend process for the loops.** The SLA loop, cert loop, and on-overdue action dispatch must run in exactly one process; multiple workers would double-process. (Same as the existing email/WA loops.)
- **`connect_db()` index creation does not run at startup** in this codebase — code that needs uniqueness uses a deterministic `_id` or conditional atomic update instead of relying on a custom index. Don't add logic that assumes those indexes exist.
- **Indian names**: certificates render Latin/transliterated names fine; native Devanagari would need `libraqm` on the VPS (Linux-easy) — add only if required.
